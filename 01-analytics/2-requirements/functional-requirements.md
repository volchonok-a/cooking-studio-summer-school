# Функциональные требования

**Этап 2.** Что система должна делать (функции, поведение).

Не путать с нефункциональными требованиями (NFR).

Приоритеты по MoSCoW: Must — обязательно к запуску; Should — желательно; Could — приятное дополнение; Won't — вне скоупа текущей поставки.

**О нумерации.** ID требований (FR-NN) стабильны для трассировки. Нумерация сквозная; пропуски допустимы, если требование было исключено при сужении скоупа. Занятые номера не переиспользуются.

## Границы скоупа (клиентское веб-приложение + API)

Скоуп — клиентское веб-приложение (десктоп) и API для него. Функции шефа и владельца/администратора обеспечиваются существующей инфраструктурой и здесь не описываются.

**Следствия для скоупа:**

- Слоты, программы и шефы приходят из существующего бэкенда через API. Экранов создания и редактирования расписания в клиентском приложении нет.
- Программы и шефы доступны клиенту как справочные (read-only) данные — клиент видит их в карточке слота, но не управляет ими.
- Бэкенд — black-box источник истины. Гарантия «0 двойных броней» и атомарность операций обеспечиваются на его стороне; клиент полагается на контракт API.

| ID | Требование | Приоритет | Источник |
| --- | --- | --- | --- |
| FR-09 | Система должна позволять авторизованному клиенту создать бронь на выбранный слот. Бэкенд атомарно проверяет наличие мест; при отсутствии мест бронь отклоняется с понятным сообщением. | Must | R-004; бриф |
| FR-10 | Система должна позволять клиенту при записи выбрать вариант экипировки: «Своя» или «Прокат (фартук + ножи)». Лимита на прокатные наборы нет. | Must | Ответ №5; описание домена |
| FR-11 | Система должна при выборе проката экипировки добавлять к итоговой сумме доплату за прокат (тариф из API). | Must | Ответ №5; описание домена |
| FR-12 | Система должна требовать обязательного заполнения поля «Аллергии и особенности питания» при бронировании. Допустимо значение «Нет / не беспокоят». При пустом поле кнопка подтверждения неактивна. | Must | Ответ №3; описание домена |
| FR-13 | Система должна позволять клиенту одной записью забронировать несколько мест (от 1 до числа свободных на слоте) — для себя и гостей. | Must | бриф; описание домена |
| FR-14 | Система должна отображать итоговую стоимость до подтверждения брони: базовая цена × количество мест, доплата за прокат, скидка 5% (если применима), итого. | Must | Ответ №1, №4; описание домена |
| FR-15 | Система должна отображать подпись «Оплата на месте (наличные / перевод на карту)» на форме брони и экране подтверждения. Онлайн-оплата в MVP не реализуется. | Must | Ответ №1; описание домена |
| FR-16 | Система должна применять скидку 5% автоматически для клиентов со статусом «Постоянный клиент» (>5 посещений). | Should | Ответ №4; описание домена |
| FR-17 | Система должна после успешной брони показывать экран подтверждения с деталями: дата/время, программа, шеф, число мест, экипировка, аллергии, итоговая сумма, статус «Ожидает оплаты». | Must | бриф; описание домена |

**Граница со серверной интеграцией (R-004).** Детали серверной реализации брони (транзакции, SLA, маппинг полей) — вне скоупа клиентского приложения. Существующий бэкенд — black-box источник истины; атомарность операций и гарантия «0 двойных броней» обеспечиваются на его стороне. Граница закрывается контрактом API.

## Мои бронирования и отмены (клиент)

| ID | Требование | Приоритет | Источник |
| --- | --- | --- | --- |
| FR-18 | Система должна показывать клиенту список его бронирований с фильтрами/вкладками: «Предстоящие», «Прошедшие», «Отменённые». Для каждой брони отображаются: дата/время, программа, шеф, статус, число мест, экипировка, аллергии. | Must | бриф; описание домена |

| FR-20 | Система должна при отмене менее чем за 12 часов до старта (поздняя отмена) показывать предупреждение о блокировке клиента на 7 дней. После подтверждения — бронь отменена, клиент заблокирован. | Must | Ответ №2; описание домена |
| FR-21 | Система должна при неявке клиента (no-show) отображать статус «Не явился» и уведомлять клиента о блокировке на 7 дней. Фиксация no-show — на стороне бэкенда. | Must | Ответ №2; описание домена |
| FR-22 | Система должна при отмене слота студией (форс-мажор, R-008) переводить бронь в статус «Отменён студией» с указанием причины. Повторная запись на этот слот запрещена. | Must | R-008; описание домена |

**Политика отмен.** Граница «ранней» отмены — ровно за 12 часов до старта включительно. Решение о типе отмены и применении блокировки принимает сервер как источник истины по времени и статусу.
