# Бизнес-требования

> Этап 2. Зачем бизнесу продукт: цели, метрики успеха, ограничения верхнего уровня.

**Cooking Studio — клиентское веб-приложение и API для него.** Бизнес-цели, обслуживаемые существующей инфраструктурой (интерфейс шефа, администратора, финансовая отчётность), в этот список не входят.

**О нумерации требований.** ID требований (BR-NN, M-NN) сохраняются стабильными для трассировки. В нумерации есть пропуски — это следствие осознанного сужения скоупа (часть ранее заведённых требований вынесена в развитие/Phase 2 или относится к существующей инфраструктуре). Нумерация намеренно не нормализуется, чтобы не ломать связи в таблице-источнике и смежных документах.

**Роли в текущей поставке.** В текущую поставку входит только роль **Клиент** (самостоятельная запись на мастер-классы через веб-приложение). Интерфейсы шефа и администратора — **вне скоупа текущей поставки**: они обеспечиваются существующей инфраструктурой. Это осознанное решение в рамках MVP-подхода: продукт фокусируется на клиентском сценарии, который снимает основную боль (хаос записи по телефону и двойные брони), а серверные/ролевые функции остаются на стороне существующего бэкенда.

## Бизнес-требования

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| BR-1 | Устранить ручную путаницу и двойные брони, возникающие при ведении записи по телефону и в мессенджерах. Учёт записей должен вестись в единой системе, исключающей конфликты по местам. | Must | Бриф (письмо владельца, п. о хаосе в записи); Описание домена → «Цель клиентского приложения» |
| BR-2 | Дать клиентам возможность самостоятельно записываться на мастер-классы онлайн без ручного посредничества администратора («люди сами записываются, а я только смотрю»). | Must | Бриф (письмо владельца); Описание домена → «Цель клиентского приложения» |
| BR-3 | Снизить простой мест из-за поздних отмен за счёт правила отмены за 48 часов и блокировки клиентов при поздних отменах/no-show (контроль без отталкивания клиентов — «ограничить, но не злить»). | Must | Бриф (место простаивает при отмене за день); Ответы заказчика № 7–8; Описание домена → «Политика отмен» |
| BR-4 | Обеспечить учёт аллергий и диетических ограничений клиентов при записи, чтобы шеф мог подготовиться к классу. | Must | Бриф (безопасность гостей); Ответы заказчика № 12; Описание домена → «Бизнес-правила» |
| BR-5 | Запустить продукт к началу следующего сезона (жёсткий срок ~2 месяца) при ограниченном бюджете, начав с минимально необходимого набора функций (MVP-подход). | Must | Бриф (сроки и бюджет); Ответы заказчика № 20; Описание домена → «Проект», «Границы скоупа» |
| BR-6 | На старте оплата остаётся офлайн (наличные / перевод на месте); продукт лишь фиксирует запись и показывает цену. Онлайн-оплата — за рамками скоупа, заложить в развитие. | Must | Бриф (наличка/перевод, не уверены в онлайн-оплате); Ответы заказчика № 9, 11; Описание домена → «Ограничения» |
| BR-7 | Клиентское приложение — веб (desktop-first с базовой адаптацией); взаимодействует с существующим бэкендом через API. | Must | Ответы заказчика № 19; Описание домена → «Технические/платформенные» |
| BR-8 | Отмечать постоянных клиентов (5+ посещений) скидкой 5% для повышения удержания. | Should | Бриф (хочу отмечать постоянных); Ответы заказчика № 15; Описание домена → «Программа лояльности» |

## Метрики успеха

> Целевые значения — разумные ориентиры для MVP (как в исходниках: точные пороги фиксируются как допущение, при наличии данных уточняются).

| ID | Метрика | Целевой ориентир | Связь с требованиями |
| :-- | :-- | :-- | :-- |
| M-1 | Доля записей, сделанных клиентами онлайн самостоятельно (от общего числа записей за сезон). | ≥ 70% к концу первого сезона | BR-2 |
| M-2 | Число двойных броней / конфликтов по местам. | 0 (полное устранение) | BR-1 |
| M-3 | Простой мест из-за поздних отмен (доля поздних отмен от записанных). | Снижение относительно текущего уровня (целевой ориентир ≤ 10%) | BR-3 |
| M-4 | Готовность к старту сезона. | Запуск в срок ~2 месяца с критичным набором функций (просмотр классов, запись, отмены) | BR-5 |
| M-5 | Доля клиентов, указавших аллергии при записи. | 100% обязательное заполнение | BR-4 |

## Ограничения верхнего уровня

- **Срок и бюджет.** Жёсткая цель — начало следующего сезона (~2 месяца); бюджет ограничен → строго MVP-подход, «навороты» откладываются. (Бриф; Ответы заказчика № 20)
- **Оплата офлайн на старте.** Онлайн-оплата вне границ скоупа. (Ответы заказчика № 9, 11)
- **Desktop-first веб-приложение.** Клиентское приложение — веб-приложение с desktop-first подходом и базовой responsive-адаптацией; данные получает из существующего бэкенда через API. Нативная мобильная разработка или PWA вынесены за скоуп текущей итерации, чтобы сфокусироваться на проработке бизнес-логики и клиентского пути. (Ответы заказчика № 19)
- **Вне скоупа.** Функции шефа/администратора, CRUD расписания и админка обеспечиваются существующей инфраструктурой. Онлайн-оплата, программа лояльности, система рейтингов шефов — направления развития. (Описание домена → «Границы скоупа»)
- **Программа лояльности — Phase 2 (подтверждено заказчиком).** Скидка для постоянных клиентов осознанно вынесена в Phase 2 и не входит в MVP. Заказчик ранее называл функцию желаемой, поэтому она не отбрасывается, а планируется к реализации после запуска критичного набора MVP (BR-5) — это сознательное сужение скоупа ради попадания в жёсткий срок начала сезона. **Перенос согласован с заказчиком**. (Бриф; Описание домена → «Границы скоупа»)
