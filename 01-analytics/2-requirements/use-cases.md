# Use Cases

> Этап 2. Сценарии использования клиентского веб-приложения.
> **Скоуп:** клиентское веб-приложение (десктоп) и API для него. Роль «Клиент».
> **Вне скоупа:** функции шефа, администратора, CRUD расписания — обеспечиваются существующей инфраструктурой.

## Акторы

| Актор | Описание | В скоупе |
|-------|----------|----------|
| **Клиент** | Гость студии, записывается на кулинарные классы через десктопное веб-приложение | ✅ Да |
| **Система (бэкенд)** | Black-box источник истины: обеспечивает атомарность броней, расчёт блокировок, хранение данных | ✅ Да (как внешний актор) |
| ~~Шеф / Администратор~~ | Управляют расписанием через существующую инфраструктуру | ❌ Вне скоупа |

## UC-1: Регистрация и вход

**ID:** UC-1
**Название:** Регистрация и авторизация клиента
**Акторы:** Клиент, Система
**Описание:** Клиент регистрируется или входит в аккаунт для доступа к бронированию.
**Предусловия:** Клиент открыл приложение, не авторизован.
**Постусловия:** Клиент авторизован; сессия установлена.

**Основной поток:**
1. Клиент открывает страницу авторизации.
2. Клиент выбирает способ входа: Email + пароль **или** OAuth (Google / Яндекс).
3. Клиент вводит данные учётной записи или нажимает кнопку OAuth-провайдера.
4. Система отправляет запрос на бэкенд.
5. Бэкенд проверяет данные и возвращает JWT access + refresh токены.
6. Система сохраняет токены в httpOnly-куки (NFR-14).
7. Система выполняет редирект на главный экран со слотами.

**Альтернативные потоки:**
- **3а.** Клиент выбирает OAuth-провайдера:
  - 3а.1. Система перенаправляет клиента на страницу OAuth-провайдера.
  - 3а.2. Клиент авторизуется у провайдера.
  - 3а.3. Провайдер возвращает код авторизации в систему.
  - 3а.4. Система обменивает код на токены у бэкенда.
  - 3а.5. Переход к шагу 6 основного потока.

**Исключительные потоки:**
- **5а.** Бэкенд возвращает ошибку «Неверный Email или пароль»:
  - 5а.1. Система показывает понятное сообщение об ошибке.
  - 5а.2. Клиент остаётся на странице авторизации.
  - Конец.
- **5б.** OAuth-провайдер недоступен (NFR-10):
  - 5б.1. Система показывает сообщение «Сервис временно недоступен, попробуйте Email + пароль».
  - 5б.2. Клиент возвращается к выбору способа входа.
  - Конец.

**Бизнес-правила:**
- SMS-авторизация в MVP не поддерживается (FR-02).
- Токены хранятся только в httpOnly-куки, не в localStorage (NFR-14).

**FR:** FR-01, FR-02
**NFR:** NFR-3, NFR-10, NFR-14
**US:** US-1

## UC-2: Просмотр профиля и статуса лояльности

**ID:** UC-2
**Название:** Просмотр профиля и счётчика посещений
**Акторы:** Клиент, Система
**Описание:** Клиент видит счётчик пройденных классов и статус «Постоянный клиент».
**Предусловия:** Клиент авторизован.
**Постусловия:** Клиент видит свой статус лояльности.

**Основной поток:**
1. Клиент открывает раздел «Профиль».
2. Система запрашивает данные профиля у бэкенда.
3. Бэкенд возвращает данные: счётчик посещений, статус.
4. Система отображает счётчик посещений.
5. Если посещений > 5 — система отображает бейдж «Постоянный клиент» (FR-04).
6. При следующем бронировании скидка 5% применяется автоматически (FR-16).

**Исключительные потоки:**
- **3а.** Бэкенд недоступен:
  - 3а.1. Система показывает сообщение «Не удалось загрузить профиль».
  - Конец.

**FR:** FR-03, FR-04
**US:** US-2

## UC-3: Просмотр списка слотов

**ID:** UC-3
**Название:** Просмотр списка слотов на ближайшие 7 дней
**Акторы:** Клиент, Система
**Описание:** Клиент видит список доступных слотов, сгруппированных по дате.
**Предусловия:** Клиент открыл главный экран.
**Постусловия:** Клиент ознакомился со слотами.

**Основной поток:**
1. Клиент открывает главный экран.
2. Система запрашивает слоты за период `[now; now + 7 дней]` (R-027).
3. Бэкенд возвращает список слотов.
4. Система группирует слоты по дате, внутри — сортирует по времени старта.
5. Система отображает список. Для каждого слота: дата/время, программа, шеф, цена, всего/свободно мест.

**Альтернативные потоки:**
- **5а.** Слотов нет (empty state):
  - 5а.1. Система отображает сообщение «Пока нет доступных классов».
  - 5а.2. Система предлагает расширить диапазон дат через фильтр (FR-08).
  - Конец.

**Исключительные потоки:**
- **3а.** Бэкенд недоступен или таймаут:
  - 3а.1. Система показывает сообщение «Не удалось загрузить расписание».
  - 3а.2. Если данные есть в кэше — система отображает их с пометкой «Данные могли устареть» (NFR-9).
  - Конец.

**FR:** FR-05, FR-08
**NFR:** NFR-4 (p95 < 2.0 с), NFR-9
**US:** US-3

## UC-4: Фильтрация слотов

**ID:** UC-4
**Название:** Фильтрация списка слотов
**Акторы:** Клиент, Система
**Описание:** Клиент фильтрует слоты по дате, программе, шефу, наличию мест.
**Предусловия:** Клиент на странице списка слотов.
**Постусловия:** Клиент нашёл подходящий слот.

**Основной поток:**
1. Клиент нажимает на кнопку «Фильтры».
2. Система открывает панель фильтров.
3. Клиент выбирает фильтры: диапазон дат, программа, шеф, наличие мест.
4. Фильтры комбинируются (AND-логика).
5. Клиент нажимает «Применить».
6. Система запрашивает слоты с учётом фильтров.
7. Бэкенд возвращает отфильтрованный список.
8. Система обновляет список слотов.
9. Активные фильтры отображаются клиенту (можно сбросить).

**Альтернативные потоки:**
- **8а.** Нет результатов:
  - 8а.1. Система отображает empty state «По вашему запросу ничего не найдено».
  - Конец.

**Исключительные потоки:**
- **7а.** Бэкенд недоступен:
  - 7а.1. Система показывает сообщение об ошибке.
  - Конец.

**FR:** FR-06
**US:** US-4

## UC-5: Просмотр карточки слота

**ID:** UC-5
**Название:** Просмотр детальной карточки слота
**Акторы:** Клиент, Система
**Описание:** Клиент изучает все параметры слота перед записью.
**Предусловия:** Клиент выбрал слот в списке.
**Постусловия:** Клиент готов к бронированию.

**Основной поток:**
1. Клиент нажимает на слот в списке.
2. Система открывает карточку слота.
3. Система запрашивает детали слота у бэкенда.
4. Бэкенд возвращает данные.
5. Система отображает: программу/меню, длительность (~3 ч), шефа (имя, рейтинг), стоимость, стоимость проката экипировки, число свободных мест, адрес студии.
6. Кнопка «Записаться» активна только при наличии свободных мест.

**Исключительные потоки:**
- **4а.** Слот не найден или удалён:
  - 4а.1. Система показывает сообщение «Слот больше недоступен».
  - 4а.2. Клиент возвращается к списку слотов.
  - Конец.

**FR:** FR-07
**NFR:** NFR-4 (p95 < 2.0 с)
**US:** US-5

## UC-6: Бронирование слота

**ID:** UC-6
**Название:** Бронирование места на кулинарный класс
**Акторы:** Клиент, Система
**Описание:** Клиент бронирует одно или несколько мест на слот с выбором экипировки и указанием аллергий.
**Предусловия:** Клиент авторизован; не заблокирован; слот имеет свободные места.
**Постусловия:** Бронь создана; клиент получил подтверждение.

**Основной поток:**
1. Клиент нажимает «Записаться» в карточке слота.
2. Система открывает форму бронирования.
3. Клиент выбирает количество мест (1…N, где N ≤ свободных мест).
4. Клиент выбирает экипировку: «Своя» или «Прокат (фартук + ножи)».
5. При выборе проката — система добавляет доплату к итогу (FR-11).
6. Клиент заполняет обязательное поле «Аллергии и особенности питания» (допустимо «Нет / не беспокоят»).
7. Система отображает итоговую стоимость: базовая цена × кол-во мест + прокат − скидка 5% (если постоянный клиент, FR-16).
8. Система отображает подпись: «Оплата на месте (наличные / перевод на карту)» (FR-15).
9. Клиент нажимает «Подтвердить бронь».
10. Система отправляет `createBooking` с `Idempotency-Key` (NFR-9).
11. Бэкенд атомарно проверяет наличие мест и создаёт бронь (защита от овербукинга, NFR-7).
12. Бэкенд возвращает подтверждение.
13. Система показывает экран подтверждения с деталями брони и статусом «Ожидает оплаты» (FR-17).
14. Запись появляется в «Моих бронированиях» (FR-18).

**Альтернативные потоки:**
- **6а.** Клиент указывает «Нет / не беспокоят» в поле «Аллергии»:
  - 6а.1. Система принимает значение как валидное.
  - 6а.2. Переход к шагу 7 основного потока.

**Исключительные потоки:**
- **6а.** Поле «Аллергии» пустое (FR-12):
  - 6а.1. Кнопка «Подтвердить бронь» неактивна.
  - 6а.2. Система показывает подсказку «Укажите аллергии или 'Нет / не беспокоят'».
  - Конец.
- **11а.** Бэкенд возвращает ошибку `slot_full` (нет свободных мест):
  - 11а.1. Система показывает сообщение «К сожалению, места закончились. Выберите другой слот».
  - Конец.
- **11б.** Клиент заблокирован (FR-29):
  - 11б.1. Система показывает сообщение «Записи недоступны до <дата>».
  - 11б.2. Все кнопки бронирования неактивны.
  - Конец.
- **10а.** Таймаут сети или неопределённый результат (NFR-9):
  - 10а.1. Клиент повторяет запрос с тем же `Idempotency-Key`.
  - 10а.2. Бэкенд возвращает результат первой операции (бронь не дублируется).
  - 10а.3. Переход к шагу 12 основного потока.

**Бизнес-правила:**
- Оплата только на месте (BR-6).
- Лимита на прокатные наборы нет (Ответ №5).
- Аллергии — обязательное поле (PII, NFR-11).
- Бронь создаётся атомарно на бэкенде (R-004).

**FR:** FR-09, FR-10, FR-11, FR-12, FR-13, FR-14, FR-15, FR-16, FR-17
**NFR:** NFR-2 (≤3 экрана), NFR-4 (p95 < 1.0 с), NFR-7 (нет овербукинга), NFR-9 (Idempotency-Key), NFR-11 (аллергии — PII)
**US:** US-6, US-7, US-8, US-9, US-10

**Акторы:** Клиент, Система
**Описание:** Клиент видит предстоящие, прошедшие и отменённые брони.
**Предусловия:** Клиент авторизован.

**Альтернативные потоки:**
- **5а.** Клиент находится офлайн (NFR-9):

**Исключительные потоки:**
- **3а.** Бэкенд недоступен:
  - 3а.1. Система показывает сообщение об ошибке.
  - Конец.

## UC-8: Отмена бронирования клиентом

**ID:** UC-8
**Название:** Отмена записи клиентом
**Акторы:** Клиент, Система
**Описание:** Клиент отменяет предстоящую бронь; в зависимости от срока — с блокировкой или без.
**Предусловия:** У клиента есть предстоящая бронь; клиент не заблокирован.
**Постусловия:** Бронь отменена; при поздней отмене — клиент заблокирован на 7 дней.

**Основной поток (ранняя отмена, ≥12 ч):**
1. Клиент нажимает «Отменить» у брони в «Моих бронированиях».
2. Система проверяет время до старта (≥12 часов).
3. Система показывает подтверждение отмены: «Вы уверены, что хотите отменить бронь?».
4. Клиент подтверждает.
5. Система отправляет `cancelBooking` на бэкенд.
6. Бэкенд меняет статус на «Отменено клиентом», место освобождается.
7. Блокировка не применяется.
8. Система обновляет список бронирований.

**Альтернативные потоки (поздняя отмена, <12 ч):**
- **2а.** Время до старта < 12 часов (FR-20):
  - 2а.1. Система показывает предупреждение: «Отмена менее чем за 12 часов приведёт к блокировке на 7 дней. Продолжить?».
  - 2а.2. Клиент подтверждает отмену несмотря на предупреждение.
  - 2а.3. Система отправляет `cancelBooking`.
  - 2а.4. Бэкенд меняет статус и применяет блокировку на 7 дней.
  - 2а.5. Клиент получает push-уведомление о блокировке (FR-26).
  - 2а.6. Переход к шагу 8 основного потока.

**Исключительные потоки:**
- **6а.** Бэкенд возвращает ошибку (бронь уже отменена или не найдена):
  - 6а.1. Система показывает сообщение «Не удалось отменить бронь».
  - Конец.

**Бизнес-правила:**
- Граница «ранней» отмены — ровно 12 часов до старта (BR-3).
- Решение о типе отмены принимает сервер (FR-19, FR-20).
- При no-show (неявке) бэкенд фиксирует статус «Не явился» и блокирует клиента на 7 дней (FR-21).

**FR:** FR-19, FR-20, FR-21
**NFR:** NFR-4 (p95 < 1.0 с)
**US:** US-12

## UC-9: Оценка шефа

**ID:** UC-9
**Название:** Оценка шефа после посещения класса
**Акторы:** Клиент, Система
**Описание:** Клиент ставит оценку шефу (1–5 звёзд) после прошедшего класса.
**Предусловия:** Класс состоялся (статус «Состоялся»); клиент авторизован; оценка ещё не ставилась.
**Постусловия:** Оценка сохранена; рейтинг шефа обновлён.

**Основной поток:**
1. Клиент открывает прошедшую бронь со статусом «Состоялся».
2. Система отображает форму оценки (1–5 звёзд).
3. Клиент выбирает оценку.
4. Клиент нажимает «Отправить».
5. Система отправляет оценку на бэкенд.
6. Бэкенд сохраняет оценку и пересчитывает рейтинг шефа.
7. Система показывает подтверждение «Спасибо за оценку!».
8. Рейтинг шефа обновляется в карточках слотов (FR-24).

**Исключительные потоки:**
- **1а.** Класс не в статусе «Состоялся»:
  - 1а.1. Форма оценки недоступна.
  - Конец.
- **1б.** Оценка уже была поставлена:
  - 1б.1. Система показывает «Вы уже оценили этот класс».
  - Конец.

**Бизнес-правила:**
- Одна оценка на бронь (FR-23).
- Текстовые комментарии в MVP не реализуются.

**FR:** FR-23, FR-24
**US:** US-14

## UC-10: Получение уведомлений

**ID:** UC-10
**Название:** Получение системных уведомлений (push + in-app)
**Акторы:** Клиент, Система
**Описание:** Клиент получает push-уведомления и внутриприложенные уведомления о важных событиях.
**Предусловия:** Клиент авторизован.
**Постусловия:** Клиент получил уведомление.

**Основной поток (push-уведомления):**
1. Система генерирует событие (напоминание, отмена студией и т.д.).
2. Система отправляет push через Web Push API.
3. Клиент видит уведомление в браузере.

**Основной поток (in-app уведомления):**
1. В шапке приложения отображается иконка колокольчика с бейджем непрочитанных (FR-28).
2. Клиент нажимает на колокольчик.
3. Система открывает список событий: подтверждение брони, отмена клиентом, отмена студией, окончание блокировки.
4. При прочтении — бейдж уменьшается.

**Типы уведомлений:**
- Напоминание о классе за ~24 часа (FR-25).
- Отмена класса студией с причиной (FR-26).
- Блокировка после поздней отмены / no-show.
- Подтверждение брони.

**Исключительные потоки:**
- **2а.** Разрешение на push отозвано (NFR-16):
  - 2а.1. Отправка прекращается без ошибки для пользователя.
  - 2а.2. Уведомление сохраняется в in-app (колокольчик).
  - Конец.

**Бизнес-правила:**
- Только Web Push + in-app. SMS/Email — вне скоупа MVP.

**FR:** FR-25, FR-26, FR-27, FR-28
**NFR:** NFR-15, NFR-16
**US:** US-13, US-15, US-16
