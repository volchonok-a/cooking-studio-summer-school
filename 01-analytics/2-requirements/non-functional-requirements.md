# Нефункциональные требования

> Этап 2. Как система должна работать: производительность, надёжность, безопасность,
> доступность, удобство.

> **Скоуп — клиентское веб-приложение (десктоп) и API для него.** НФТ относятся к
> клиентскому приложению; функции шефа/владельца/админки обеспечиваются существующей
> инфраструктурой и в скоуп не входят.

> **О нумерации ID.** Текущая нумерация сквозная (NFR-1, NFR-2, ...). В будущих версиях
> документа при удалении или изменении требований существующие ID сохраняются, чтобы не
> ломать трассируемость; новые требования получают следующие свободные номера.

> **Целевая платформа.** Продукт — веб-приложение, рассчитанное на использование
> с компьютера (десктопные браузеры: Chrome, Firefox, Safari, Edge последних 2 версий).
> Требования к хранению токена, сессии, уведомлениям и доступности сформулированы
> исходя из этого решения (httpOnly-куки, Web Push API, HTTPS, клавиатурная навигация,
> hover-состояния).

## Удобство использования

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| NFR-1 | Интерфейс клиента спроектирован для использования с компьютера (десктоп-first): широкие макеты (основной breakpoint ≥ 1280 px), hover-состояния для интерактивных элементов, поддержка клавиатурной навигации (Tab, Enter, Esc), минимум полей и шагов. | Высокий | бриф; описание домена |
| NFR-2 | Запись на слот выполняется за минимальное число шагов — ориентир ≤3 экранов до подтверждения брони, чтобы клиент мог записаться быстро и без обучения. | Высокий | бриф; описание домена |
| NFR-3 | Лёгкая регистрация: Email + пароль или OAuth (Google, Яндекс). SMS-авторизация исключена для упрощения MVP. Порог входа для клиента минимален. | Высокий | Ответ №6; описание домена |

> **Решение по UX.** Клиент пользуется приложением с компьютера, обычно в спокойной
> обстановке (дом, офис). Поэтому приоритет — широкие макеты, hover-состояния и
> клавиатурная навигация, а не touch-target.

## Производительность

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| NFR-4 | Целевые метрики при «нормальной сети» (проводной интернет / Wi-Fi, ≥10 Мбит/с): получение списка слотов и открытие карточки слота — **p95 < 2.0 с**; создание брони (`createBooking`) и отмена брони (`cancelBooking`) — **p95 < 1.0 с**. Метрики измеряются на стороне клиента от инициации запроса до готовности данных к отображению. | Высокий | бриф; описание домена |
| NFR-5 | Система сохраняет приемлемую отзывчивость при пиковой нагрузке в выходные, когда одновременно записывается наибольшее число клиентов. | Высокий | бриф; описание домена |
| NFR-6 | Расчётный пик — выходные в сезон, оценочно до ~50 одновременных активных пользователей. Под пиком сохраняются метрики NFR-4. **Сценарий нагрузочного теста на гонку бронирований:** N (≈20–50) клиентов одновременно вызывают `createBooking` на один слот с числом мест, заведомо превышающим остаток; проверяется, что подтверждается ровно столько броней, сколько было свободных мест, остальным возвращается корректная ошибка (`slot_full`), двойных броней и овербукинга нет (см. NFR-7), p95 успешного ответа удерживается в пределах NFR-4. | Высокий | бриф; описание домена; NFR-7 |

> **Решение по производительности.** Метрики ужесточены по сравнению с mobile-first
> сценарием: десктоп обычно работает по более быстрому каналу (проводной интернет /
> Wi-Fi ≥10 Мбит/с), поэтому целевые p95 снижены. Бэкенд — black-box (R-004), его SLA
> не гарантируем; клиентская часть должна быть отзывчивой даже при медленном API.

## Надёжность / Целостность данных

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| NFR-7 | При одновременной записи нескольких клиентов на один слот исключаются двойные брони и овербукинг: количество занятых мест никогда не превышает лимитов слота (12 для обычных классов, 8 для классов с духовкой). Это критическое требование (главная боль заказчика). Гарантия обеспечивается на стороне бэкенда (black-box, R-004); клиент корректно обрабатывает отказ при отсутствии мест. | Высокий (Must) | бриф; описание домена; R-004 |
| NFR-8 | Состояние записей и свободных мест остаётся согласованным при сбоях и параллельных операциях (отмена освобождает место корректно, без потери или задвоения данных). Контроль на стороне бэкенда. | Высокий | бриф; описание домена |
| NFR-9 | Просмотр уже загруженных данных (списка слотов, своих броней) доступен офлайн из кэша (localStorage/IndexedDB) с явной пометкой «данные могли устареть». Мутации (создание/отмена брони) офлайн **запрещены** — показывается понятное сообщение о необходимости подключения. Единый паттерн Error/Retry для всех сетевых ошибок. Таймаут сетевого запроса — ~10 с. При неопределённом результате создания брони (таймаут/обрыв) клиент повторяет запрос с тем же `Idempotency-Key`, что исключает дубли. | Средний | R-004; R-022; описание домена |

> **Граница со серверной интеграцией (R-004).** Детали серверной реализации —
> транзакционность, SLA, маппинг сущностей на внутренние модели бэкенда — вне скоупа
> клиентского приложения. Существующий бэкенд — black-box источник истины: гарантия
> «0 двойных броней» обеспечивается на его стороне. Со стороны клиента граница
> закрывается контрактом API.

## Доступность (Availability)

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| NFR-10 | Целевая доступность — **99%** в рабочие часы студии в течение сезона. Окно измерения и все временные правила трактуются в **локальной таймзоне студии** (по умолчанию — единая локальная зона студии). Конкретные рабочие часы студии, границы сезона и таймзона фиксируются как **обязательный pre-launch config** с владельцем (владелец продукта / эксплуатация); **критерий приёмки:** значения заданы и проверены до запуска. Из расчёта доступности **исключаются**: плановые работы (с предварительным уведомлением) и недоступность внешней инфраструктуры (OAuth-провайдеры Google/Яндекс, push-сервисы). **Деградация:** при отказе OAuth-провайдера — fallback на Email+пароль; при отказе push-сервиса — бронирование и отмена работают без напоминаний. | Средний | описание домена |

> **Решение по часовым зонам.** Студия работает в одной локации (лофт на бывшем заводе).
> Все временные правила (12 часов до отмены, рабочие часы, сезон) трактуются в локальной
> таймзоне студии. Это явное допущение, зафиксированное как pre-launch config.

## Безопасность / Приватность

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| NFR-11 | Персональные данные клиентов (Email, имя) защищены от несанкционированного доступа; хранятся и передаются безопасно. Поле «Аллергии и особенности питания» — PII (медицинские данные), передаётся только по HTTPS, не логируется в клиентских логах. | Высокий | описание домена; Ответ №3 |
| NFR-12 | Базовый набор требований к ПДн для учебного проекта (Email, имя, аллергии): (1) явное согласие пользователя и доступная политика конфиденциальности до отправки данных; (2) удаление/анонимизация данных пользователя по запросу на удаление аккаунта, включая анонимизацию связанных броней; (3) передача всех ПДн только по TLS (см. NFR-14); (4) маскирование чувствительных данных в UI и в логах (запрет логирования полного Email, аллергий, токенов); (5) базовый retention: прошедшие/отменённые брони хранятся **12 месяцев** после события (завершения/отмены класса), далее анонимизируются. **Вне скоупа MVP (Phase 2):** полный комплаенс 152-ФЗ — экспорт данных субъекта, журнал/аудит доступа администраторов, формализованные юридические основания обработки. | Высокий | R-017; Ответ №3; описание домена |

> **Решение по ПДн.** Аллергии — медицинские данные, это PII. В MVP — базовый набор
> мер (согласие, маскирование, retention 12 мес.). Полный комплаенс 152-ФЗ — в Phase 2,
> после согласования с юристами.

## Безопасность / Сессия

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| NFR-14 | Сессия — JWT с парой токенов: короткоживущий **access** (срок жизни ~15 мин) и долгоживущий **refresh** (срок жизни ~30 дней). Access автоматически обновляется по refresh без участия пользователя; повторный вход требуется только при истечении/отзыве refresh. Оба токена хранятся в httpOnly-куки (не в localStorage), защищены от XSS. **401-flow:** при `401` от API клиент пытается обновить access по refresh; при неуспехе — очищает токены и переводит пользователя на экран авторизации. Весь обмен с API — только по TLS (HTTPS, TLS 1.2+). | Высокий | R-016; описание домена |

> **Решение по сессии.** httpOnly-куки — стандарт для веб-приложений, защита от XSS.
> 401-flow с попыткой обновить access по refresh — чтобы клиент не терял сессию при
> кратковременном истечении access-токена.

## Уведомления

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| NFR-15 | Напоминания о записи доставляются своевременно (заблаговременно до старта класса), чтобы снизить неявки. Доставка обеспечивается Web Push API. | Средний | бриф; Ответ №3 |
| NFR-16 | Канал уведомлений MVP — Web Push API + in-app уведомления (иконка колокольчика). Push-токен регистрируется на бэкенде после получения системного разрешения; при отзыве разрешения отправка прекращается без ошибки для пользователя. Напоминание о записи отправляется за ~24 часа до старта класса (согласовано с заказчиком). SMS/Email-уведомления — вне скоупа MVP. | Средний | R-006; Ответ №3; описание домена |

> **Решение по уведомлениям.** Только Web Push + in-app. SMS/Email — в следующих версиях,
> если потребуется. Push-токен регистрируется на бэкенде, клиент не управляет им напрямую.
