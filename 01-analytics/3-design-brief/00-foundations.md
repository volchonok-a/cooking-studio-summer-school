# Foundations: основы проектирования

## 1. Платформа и технические ориентиры

- **Desktop-first веб-приложение** (Chrome, Firefox, Safari, Edge — последние 2 версии)
- **Основной breakpoint ≥ 1280 px**, рабочий диапазон 1280–1920 px
- **Не делаем:** мобильное приложение, PWA, адаптацию под узкие экраны как приоритет
- **Следствие для дизайна:** широкие макеты, hover-состояния, клавиатурная навигация (Tab/Enter/Esc) важнее крупных touch-target

## 2. Ключевые UX-требования

| Требование | Что значит для дизайна |
|---|---|
| **≤ 3 экрана до подтверждения брони** | Путь «выбрал слот → подтвердил» должен быть коротким |
| **Минимум полей и шагов** | Только обязательные поля: количество мест, экипировка, аллергии |
| **WCAG 2.1 AA** | Контраст текста, фокус-состояния, labels для screen reader |
| **Hover = Focus** | Все hover-состояния должны дублироваться фокусом (клавиатура) |
| **Понятные ошибки и empty states** | Предусмотреть все состояния списков и форм |

## 3. Доступность (A11y)

**Приёмочные критерии:**
- Контраст текста соответствует WCAG 2.1 уровня AA
- Полная поддержка клавиатурной навигации (Tab, Shift+Tab, Enter, Space, Esc)
- Все интерактивные элементы имеют labels для screen reader
- Смысл (статусы, ошибки) не передаётся одним только цветом
- Hover-состояния дублируются focus-состояниями

## 4. Локализация

- **MVP:** только русский язык
- Все строки должны быть вынесены в i18n-файлы для будущего расширения

## 5. Особые требования

### 5.1. Поле «Аллергии»
- Обязательно для заполнения (PII — медицинские данные)
- При пустом поле кнопка неактивна
- Допустимо значение «Нет / не беспокоят»

### 5.2. Офлайн-режим
- Просмотр загруженных данных — с плашкой «Данные могли устареть»
- Мутации офлайн запрещены — нужен понятный state «Нет подключения»

### 5.3. Push-уведомления
- Запрос разрешения при первом входе/брони
- Нужен UI-паттерн (модалка/баннер с объяснением, зачем)
