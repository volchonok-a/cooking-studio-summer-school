# LOGIC-006: Офлайн-режим и защита от дублей

**ID:** LOGIC-006

**Тип:** Техническая логика

**Домен:** Общий (Технический UX)

**Приоритет:** High

**Статус:** Актуален

---

## Описание
Поведение приложения при потере сети: кэширование данных для чтения, запрет мутаций, механизм Idempotency-Key для защиты от дублей броней.

### Цель
Обеспечить непрерывность UX и предотвратить двойные брони при сетевых сбоях.

### Контекст использования
Все экраны приложения.

---

## Шаги выполнения

### Шаг 0: Мониторинг соединения
- Frontend отслеживает `navigator.onLine` и события `online`/`offline`.
- При потере сети → показать глобальный баннер «Нет подключения к интернету».

### Шаг 1: Кэширование (Read)
- **Что кэшируется:** список слотов (SCR-002), детали слота (SCR-003), бронирования (SCR-006), профиль (SCR-009).
- **Где хранится:** `localStorage` или `IndexedDB`.
- **Индикатор:** жёлтая плашка «⚠️ Данные могли устареть».

### Шаг 2: Запрет мутаций (Write)
- При `navigator.onLine === false`:
  - Все кнопки «Записаться», «Отменить», «Оценить» → `disabled`.
  - При клике → toast «Нет подключения к интернету. Действие недоступно.»

### Шаг 3: Восстановление соединения
- При `navigator.onLine === true`:
  - Баннер «Нет подключения» исчезает.
  - Автоматический retry последнего неудачного запроса (если безопасно).
  - Баннер «Данные могли устареть» остаётся до следующего успешного запроса.

### Шаг 4: Idempotency-Key (NFR-5, NFR-7)
**Проблема:** при таймауте `POST /bookings` клиент может повторить запрос → риск двойной брони.

**Решение:**
1. Frontend генерирует уникальный `idempotency_key` (UUID v4) **до** отправки запроса.
2. Передаёт его в теле запроса:
   ```json
   POST /bookings
   {
     "slot_id": "...",
     "guest_count": 2,
     "equipment": "rent",
     "allergies": "...",
     "idempotency_key": "9b1deb4d-3b7d-4bad-9bdd-2b0d7b3dcb6d"
   }
   ```
3. При повторе запроса с тем же ключом бэкенд возвращает результат первой операции, не создавая новую бронь.

### Шаг 5: Таймауты (NFR-9)
- Значение: ~10 секунд.
- При таймауте → сообщение «Превышено время ожидания. Проверьте сеть и повторите.»
- Кнопка «Повторить» отправляет запрос заново (с тем же `idempotency_key`).

---

## Бизнес-правила

- **Read:** данные из кэша + плашка «Данные могли устареть».
- **Write:** мутации запрещены, кнопки `disabled`.
- **Таймаут:** 10 секунд для всех запросов.
- **Idempotency-Key:** уникальный UUID на каждую операцию создания брони.
- **0 двойных броней** гарантируется бэкендом (NFR-7).
