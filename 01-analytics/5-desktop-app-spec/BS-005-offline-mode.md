# BS-005: Офлайн-режим (Offline Mode)

**ID:** BS-005

**Тип:** Бизнес-сценарий / Состояние интерфейса

**Домен:** Общий (Технический UX)

**Приоритет:** High

**Статус:** Актуален

---

## Описание
Поведение клиентского приложения при потере сетевого соединения. Приложение должно позволять просматривать ранее загруженные данные, но запрещать мутации до восстановления связи.

### Цель
Обеспечить непрерывность пользовательского опыта (UX) при нестабильном интернете и предотвратить потерю данных или двойные брони.

### Контекст использования
Все экраны приложения.

---

## Элементы интерфейса

### Баннеры состояний

| Параметр | Значение |
|----------|----------|
| Баннер «Нет подключения» | Заметный, не перекрывает контент. Кнопки мутаций `disabled`. |
| Баннер «Данные устарели» | Менее агрессивный (жёлтый/серый). Видим, пока не произойдёт refresh. |

---

## Сценарии / Состояния

### Сценарий 1: Кэширование (Read)
**Контекст:** SCR-002, SCR-003, SCR-006, SCR-009.
**Действие:** Данные отображаются из `localStorage`/`IndexedDB`.
**Индикатор:** Жёлтая плашка «️ Данные могли устареть».

### Сценарий 2: Запрет мутаций (Write)
**Контекст:** SCR-004 (бронь), SCR-007 (отмена), SCR-008 (оценка).
**Действие:** Кнопки действий переходят в состояние `disabled`.
**Индикатор:** Toast/баннер «Нет подключения к интернету. Действие недоступно».

### Сценарий 3: Восстановление соединения
**Контекст:** Все экраны.
**Действие:** Баннер «Нет подключения» исчезает. Система автоматически повторяет последний неудачный запрос (если это безопасно).

---

## Бизнес-правила

- **Защита от дублей (Idempotency):** При каждом запросе на создание брони клиент генерирует уникальный `Idempotency-Key` (UUID). Повторный запрос с тем же ключом не создаст вторую бронь.
- **Таймауты:** ~10 секунд для всех сетевых запросов. При таймауте — сообщение «Превышено время ожидания» и кнопка «Повторить».
- Skeleton показывается только при первичной загрузке. При ошибке сети на уже загруженной странице старый контент сохраняется.
