# AI Diary — Кулинарная студия

Журнал взаимодействия с нейросетью для проекта веб-приложения кулинарной студии.

## Структура записей
Каждая запись содержит:
- **Дата/время** — когда был запрос
- **Этап** — на каком этапе работы
- **Контекст** — что мы делали перед запросом
- **Промпт** — что спросили у нейросети
- **Ответ** — краткая выжимка или полный ответ
- **Решение** — что решили делать дальше

## 1. Elicitation (Выявление требований)

### Запись 1.1 — Первый анализ брифа
**Дата:** 05.07.2026

**Контекст:** Изучили бриф от Артёма, нужно понять, какие вопросы задать заказчику.

**Промпт:** 
>"изучи бриф и сформулируй вопросы для заказчика"

**Ответ (кратко):** Нейросеть предложила 6 групп вопросов:
1. Оплата и бронирование (онлайн-оплата vs на месте)
2. Политика отмен (дедлайн бесплатной отмены)
3. Уведомления и аллергии (каналы, обязательность поля)
4. Программа лояльности (как определять постоянных клиентов)
5. Прокат экипировки (оплата, лимиты)
6. Подтверждение скоупа (только клиентское приложение)

**Решение:** Отправить вопросы заказчику (или сымитировать ответы).

**Дата:** 05.07.2026

**Контекст:** Заказчик ответил на все вопросы.

**Промпт:** 
>"давай я в формате вопрос-ответ отвечу на вопросы"

**Ответ (кратко):** Зафиксировали ответы:
- Оплата на месте, без предоплаты
- Отмена за 12 часов бесплатно
- Блокировка на неделю при no-show

**Решение:** Создать файл `qa-answers.md` с полными ответами.

### Запись 2.4 — Бизнес-требования
- **Дата:** 05.07.2026
- **Этап:** Requirements (формализация требований)
- **Контекст:** После сбора функциональных (FR) и нефункциональных (NFR) требований, а также пользовательских историй (US), нужно подняться на уровень выше и зафиксировать бизнес-цели продукта — зачем вообще бизнесу нужен этот продукт и как измерить успех.

**Промпт:**
>"теперь мне надо сделать бизнес требования. вот пример https://github.com/surfstudio/summer-school-2026/blob/feature/client/01-analysis/2-requirements/business-requirements.md. пиши маркдаун кодом"

**Ответ (кратко):** Нейросеть сгенерировала документ `business-requirements.md` со структурой:
- Бизнес-требования (BR-1 … BR-8) — верхнеуровневые цели продукта: устранение двойных броней, самостоятельная запись клиентов, снижение простоя мест, учёт аллергий, запуск к сезону, оплата офлайн, веб-приложение, программа лояльности.
- Метрики успеха (M-1 … M-5) — измеримые показатели: доля онлайн-записей (≥70%), число двойных броней (0), простой мест (≤10%), готовность к сезону (~2 месяца), доля клиентов с заполненными аллергиями (100%).
- Ограничения верхнего уровня — жёсткий срок (~2 месяца), оплата только офлайн, desktop-first веб (не мобайл), вне скоупа — шеф/админ, онлайн-оплата — Phase 2.
- Важное исправление: BR-3 — срок отмены 12 часов (а не 48, как было в первой версии), согласно ответу заказчика №2.
- Важное исправление: BR-8 — программа лояльности включена в MVP (счётчик + скидка 5%), а в Phase 2 вынесены только сложные механики бонусов. Устранено противоречие с разделом ограничений.

**Решение:** Сохранить сгенерированный документ в файл `business-requirements.md`. Проверить трассируемость: каждое BR должно ссылаться на источник (бриф, ответы заказчика, описание домена). Убедиться, что метрики покрывают все Must-требования.

### Запись 2.5 — Use Cases
- **Дата:** 05.07.2026
- **Этап:** Requirements (формализация требований)
- **Контекст:** После бизнес-требований (BR), функциональных (FR), нефункциональных (NFR) требований и пользовательских историй (US) нужно описать сценарии использования — как именно клиент взаимодействует с системой шаг за шагом, включая альтернативные и исключительные потоки. Use Cases детализируют FR и US до уровня конкретных действий пользователя и реакций системы.

**Промпт:**
>"теперь мне нужно написать use cases. вот пример https://github.com/surfstudio/summer-school-2026/blob/feature/client/01-analysis/2-requirements/use-cases.md. пиши маркдаун кодом"

**Ответ (кратко):** Нейросеть сгенерировала документ `use-cases.md` со структурой:
- Акторы — Клиент (в скоупе), Система/бэкенд (внешний актор, black-box), Шеф/Администратор (вне скоупа).
- 10 Use Cases (UC-1 … UC-10), покрывающих все ключевые сценарии клиента:
  - UC-1: Регистрация и вход (Email+пароль / OAuth)
  - UC-2: Просмотр профиля и статуса лояльности
  - UC-3: Просмотр списка слотов (7 дней)
  - UC-4: Фильтрация слотов
  - UC-5: Просмотр карточки слота
  - UC-6: Бронирование слота (с экипировкой, аллергиями, скидкой)
  - UC-7: Просмотр своих бронирований
  - UC-8: Отмена бронирования клиентом (ранняя ≥12ч / поздняя <12ч)
  - UC-9: Оценка шефа после посещения
  - UC-10: Получение уведомлений (push + in-app колокольчик)
- Для каждого UC — основной поток (нумерованные шаги 1, 2, 3…), альтернативные потоки (A1, A2, A3…), исключительные потоки (E1, E2, E3…), бизнес-правила, ссылки на FR/NFR/US.
- Матрица трассируемости Use Cases → FR / US.
- Ключевые решения: скоуп ограничен ролью «Клиент», бэкенд — black-box (R-004), политика отмен 12 часов (BR-3), оплата только офлайн (BR-6), уведомления только Web Push + in-app, Idempotency-Key (NFR-9), аллергии — PII (NFR-11).

**Решение:** Сохранить сгенерированный документ в файл `use-cases.md`. Проверить, что:
1. Все FR и US покрыты хотя бы одним Use Case (матрица трассируемости).
2. Альтернативные и исключительные потоки описаны для ключевых сценариев (бронирование, отмена, авторизация).
3. Нотация потоков соответствует примеру (A1/A2/A3 для альтернативных, E1/E2/E3 для исключительных).
4. Бизнес-правила (12 часов, 5% скидка, аллергии PII) отражены в соответствующих UC.

## 4. Design (Проектирование архитектуры и данных)

### Запись 4.1 — Разработка диаграмм последовательности API
**Дата:** 05.07.2026

**Контекст:** На основе Use Cases, пользовательских историй (User Stories) и функциональных требований (FR) необходимо спроектировать логику взаимодействия компонентов системы и контракты API в виде sequence-диаграмм, исключив отправку запросов «вслепую» и предусмотрев корректную обработку ошибок.

**Промпт:** 
>"Изучи файлы бизнес-требований (business-requirements.md), функциональных требований (functional-requirements.md), нефункциональных требований (non-functional-requirements.md) и Use Cases (use-cases.md). Спроектируй для роли Клиента ключевые сценарии взаимодействия в файле api-sequence.md с использованием диаграмм Mermaid (sequenceDiagram). Оставь следующие процессы:
>Авторизация через Email + пароль.
>Создание бронирования (createBooking): обязательно спроектируй три ветки ответов API с обработкой HTTP-статусов: 201 Created (успех), 409 Conflict (овербукинг/нет мест) и 410 Gone (слот отменен студией по форс-мажору).
>Поздняя отмена бронирования (менее чем за 12 часов до старта по BR-3): спроектируй двухэтапный процесс взаимодействия (сначала вызов GET-эндпоинта предпроверки штрафов, затем рендеринг фронтендом модального окна с предупреждением о блокировке аккаунта на 7 дней по FR-29, получение явного согласия клиента и только после этого отправка DELETE-запроса).
>Под каждой диаграммой добавь таблицу трассировки шагов в формате 'Шаг | Действие (Фронтенд / Бэкенд) | Назначение и трассировка', где свяжи каждый шаг с конкретными ID требований (FR, US, UC, BR)."

**Ответ (кратко):** Нейросеть сгенерировала структуру api-sequence.md с Mermaid-схемами, учитывающими бизнес-правило отмены за 12 часов (BR-3), двухэтапную отмену с модальным окном согласия, ветвление ответов при создании брони (201/409/410) и генерацию ключа идемпотентности, а также приложила таблицы трассировки шагов ко всем сценариям.

**Решение:** Перенести сгенерированный код в файл `api-sequence.md`.

### Запись 4.2 — Проектирование логической модели данных
**Дата:** 05.07.2026

**Контекст:** Требуется зафиксировать логическую структуру данных (Data Model) для кулинарной студии, включая сущности, связи, типы полей, а также логику клиентского кэширования в офлайне, разграничив при этом права доступа приложения (сущности black-box бэкенда).

**Промпт:** 
>"Напиши спецификацию модели данных в файле data-model.md для системы бронирования кулинарной студии. Структура должна строго соответствовать следующим разделам:
>ER-диаграмма связей (Mermaid erDiagram) для сущностей Client, Chef, Slot, Booking, Rating. Диаграмма должна быть чистой (без указания уровней доступа Read/Write в комментариях к полям). Учти типы UUID, Timestamp, поля для блокировок (is_blocked, blocked_until), поле статуса слота (status) и поле лояльности (visit_count).
>Описание сущностей и бизнес-логика полей — краткое текстовое перечисление назначения каждой сущности и детальное описание каждого её атрибута, типа данных и роли в логике приложения (например, обязательность поля allergies по FR-12 и роль visit_count в расчете скидок по FR-04).
>Локальное кэширование приложения (Frontend Storage) — пример json-структуры кэша расписания (slots_cache) на 7 дней в localStorage для поддержки офлайн-режима (NFR-9), а также описание параметров безопасности сессионных Cookies (accessToken и httpOnly/Secure/SameSite токенов refreshToken по NFR-14) и использование sessionStorage для флагов состояния профиля."

**Ответ (кратко):** Нейросеть сформировала файл data-model.md, содержащий чистую техническую ER-диаграмму сущностей кулинарной студии, детальные описания атрибутов с привязкой к логике MVP и требованиям лояльности, а также структуру JSON-кэша слотов и параметры хранения сессионных токенов безопасности.

**Решение:** Объединить блоки ER-диаграммы, текстовых сущностей, таблиц бэкенда и спецификации клиентского кэша в итоговый артефакт `data_model.md`.
